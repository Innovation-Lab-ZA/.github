# Security Policy

## Reporting a Vulnerability

Innovation Lab ZA takes security seriously. If you discover a security vulnerability in any of our projects, hackathon tools, or competition infrastructure, please report it responsibly.

### How to Report

- **Email:** innovation-lab-za@proton.me
- **Subject:** `[SECURITY] Brief description of the issue`
- Include as much detail as possible: affected project, steps to reproduce, potential impact

### What to Expect

| Step | Timeline |
|---|---|
| Acknowledgement of receipt | Within 48 hours |
| Initial assessment and confirmation | Within 5 business days |
| Fix development and testing | Varies by severity |
| Public disclosure | After fix is released |

### Scope

This policy applies to:
- All repositories under the `Innovation-Lab-ZA` GitHub organization
- Official competition and hackathon platforms we operate
- Tools and scripts used in our events

### Out of Scope

- Issues already publicly disclosed
- Vulnerabilities in third-party dependencies (report to the upstream project)
- Social engineering or phishing attempts (report to your institution's IT security)

### Recognition

We maintain a **Security Hall of Fame** for responsible disclosures. Contributors will be acknowledged (with permission) in the project's README and on our website.

---

## Secure Development Standards

All Innovation Lab ZA projects should:

- Keep dependencies up to date (enable Dependabot)
- Never commit secrets, API keys, or credentials
- Use environment variables for configuration
- Follow OWASP Top 10 best practices
- Review code for security issues before merging
