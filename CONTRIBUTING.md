# Contributing to Innovation Lab ZA

First off — **thank you for being here**. Whether you're a student, a mentor, or just someone who loves building, we're excited to have you contribute.

---

## How to Get Started

### 1. Find Something to Work On

- Browse our **[Projects](https://github.com/orgs/Innovation-Lab-ZA/projects)** tab for active initiatives
- Check the **Issues** tab on any repo for `good first issue` tags
- Join an active **hackathon** or competition team
- Propose your own idea — start a **[Discussion](https://github.com/orgs/Innovation-Lab-ZA/discussions)**

### 2. Set Up Your Environment

1. Fork the repository
2. Clone your fork locally
3. Read the repo's `README.md` for setup instructions
4. Create a new branch for your work

```bash
git checkout -b feature/your-feature-name
```

### 3. Make Your Changes

- Write clean, readable code
- Follow the project's existing style and conventions
- Add comments where the logic isn't obvious
- Write or update tests if the project has them
- **Vibe code** is welcome — but clean it up before you PR

### 4. Submit a Pull Request

1. Push your branch to your fork
2. Open a PR against the `main` branch
3. Fill out the PR template completely
4. Link any related issues
5. Tag relevant team members for review

---

## Hackathon & Competition Contributions

If you're joining for a hackathon or competition:

- Check the repo's **Hackathon Guide** (if available) for specific rules
- Teams are self-organised — communicate in the repo's Discussions
- Submissions must include a working MVP and documentation
- Code must be original or properly attributed — **no plagiarism**

---

## Style Guides

### Git Commit Messages

- Use the present tense: `Add feature` not `Added feature`
- Keep the first line under 72 characters
- Reference issues and PRs where applicable

```
feat: add leaderboard component for hackathon tracker
fix: resolve race condition in matchmaking logic
docs: update contributing guide with hackathon rules
```

### Code Style

Each repo may have its own style guide. When in doubt:
- Follow the language's community-standard style (PEP 8 for Python, Prettier for JS/TS, etc.)
- Lint before you commit
- Keep functions small and focused

## Community

- Be respectful — read our **[Code of Conduct](./CODE_OF_CONDUCT.md)**
- Help others — review PRs, answer questions, mentor new contributors
- Share what you learn — write docs, record demos, give talks
- Have fun — this is about growth, not perfection
